== Becoming a Maintainer

We welcome new maintainers! The path to maintainership:

. *Contribute regularly* - Demonstrate sustained, quality contributions over 3-6 months
. *Show expertise* - Deep knowledge in a specific area (backend, frontend, security, etc.)
. *Community engagement* - Help others, review PRs, participate in discussions
. *Alignment* - Demonstrate alignment with project values and coding standards
. *Nomination* - Current maintainer nominates you, other maintainers vote

=== Maintainer Expectations

* *Response time*: Respond to assigned issues/PRs within 7 days
* *Code review*: Review PRs in your area of expertise
* *Communication*: Participate in maintainer discussions
* *Availability*: Commit to at least 4-8 hours/month
* *Conduct*: Follow and enforce Code of Conduct

=== Maintainer Privileges

* Write access to repository
* Ability to merge PRs
* Voting rights on project decisions
* Recognition in project documentation
* Listed in this MAINTAINERS.adoc file

== Decision Making

=== Minor Decisions

* Code style, small refactors, documentation updates
* *Process*: Single maintainer approval

=== Major Decisions

* Architecture changes, new dependencies, breaking changes
* *Process*: Majority vote among active maintainers (lazy consensus)
* *Timeline*: 7 days for discussion before vote

=== Critical Decisions

* License changes, project governance, CoC violations
* *Process*: Unanimous consent among all maintainers
* *Timeline*: 14 days minimum discussion period

== Stepping Down

Maintainers can step down at any time by:

. Notifying other maintainers via email
. Creating PR to move themselves to "Emeritus" section
. Ensuring knowledge transfer for critical areas
. (Optional) Identifying potential successor
