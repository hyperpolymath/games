== Becoming a Maintainer

We welcome community members who demonstrate:

1. _Sustained Contributions_: Regular, high-quality contributions over 6+ months
2. _Technical Excellence_: Deep understanding of project architecture and goals
3. _Community Engagement_: Helpful in issues, PRs, and discussions
4. _Alignment with Values_: Commitment to ethical security research and inclusive community

=== Process

1. _Nomination_: Existing maintainer nominates candidate
2. _Evaluation_: Review contribution history and community engagement
3. _Consensus_: Existing maintainers vote (75% approval required)
4. _Onboarding_: New maintainer receives access and mentoring

=== Expectations

Maintainers are expected to:

- Review pull requests in their area within 7 days
- Respond to security issues within 48 hours
- Participate in project discussions and decisions
- Uphold the Code of Conduct
- Mentor new contributors
- Maintain areas of responsibility
- Be available for critical issues

== Maintainer Responsibilities

=== Pull Request Review

- Review PRs within 7 days
- Provide constructive feedback
- Ensure tests pass
- Check security implications
- Verify documentation updates
- Approve or request changes

=== Issue Management

- Triage new issues within 3 days
- Label appropriately
- Ask for clarification if needed
- Close duplicates or invalid issues

=== Security

- Manage security advisories

=== Releases

- Update CHANGELOG

=== Community

- Welcome new contributors
- Answer questions
- Enforce Code of Conduct
- Foster inclusive environment
- Recognize contributions

== Decision Making

=== Regular Decisions

- Pull request approvals: Any maintainer
- Issue triage: Any maintainer
- Minor changes: Lazy consensus (72 hours)

=== Major Decisions

Major decisions require consensus among maintainers:

- Architecture changes
- Breaking API changes
- New major features
- Security policy changes
- Governance changes
- Adding/removing maintainers

=== Conflict Resolution

1. Discussion among maintainers
2. Attempt to reach consensus
3. If no consensus, majority vote (75% threshold)
4. Lead maintainer has tie-breaking vote

== Stepping Down

Maintainers may step down by:

1. Notifying other maintainers
2. Completing in-progress work or transitioning to others
3. Transferring responsibilities
4. Updating this document

=== Emeritus Status

Maintainers who step down in good standing receive emeritus status:

- Listed in MAINTAINERS.md
- Recognized for contributions
- May advise on major decisions
- Can return to active status

== Inactive Maintainers

Maintainers inactive for 6+ months without notice:

1. Contacted by other maintainers
2. 30-day grace period
3. If no response, moved to emeritus status
4. Access rights adjusted accordingly
